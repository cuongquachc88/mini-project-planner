Software Architecture Document

Mini Project Planner · v1.0 · Last updated 2026-07-01

Architecture Overview

Mini Project Planner is a fully client-side, offline-first Progressive Web App. There is no backend server. All persistent data lives in PGlite — a full PostgreSQL engine compiled to WebAssembly — running entirely in the browser and persisted to IndexedDB. Google Drive sync is an optional, voluntary operation; the app works completely without it.

Core invariant: No data ever leaves the device without an explicit user action (Drive sync). The app must be 100% functional offline after initial load.

Architecture layers

  • UI: React 18 + TypeScript. Component tree rendered by React DOM.
  • State: Zustand (UI-only — filters, drag state, modal open/close). No data cache in Zustand.
  • Live data: useLiveQuery from @electric-sql/pglite-react — components re-render automatically when the DB rows they query change. Single source of truth.
  • Database: PGlite via IdbFs (IndexedDB persistence). Full PostgreSQL dialect including joins, aggregates, CTEs.
  • Routing: React Router v6 with nested layouts.
  • Build: Vite 5. @vitejs/plugin-react for HMR. vite-plugin-pwa for service worker + manifest.

Data Model

All PKs are client-generated crypto.randomUUID() text UUIDs. All tables include project_id as a foreign key — there is no cross-project data leakage.

Core entity relationships (ASCII ER)

users
  └─ project_members ─── projects
                              ├─ custom_stages
                              ├─ labels
                              ├─ epics
                              │    └─ work_items ──── work_item_labels
                              ├─ milestones │              └─ labels
                              ├─ sprints ───┘
                              │              └─ comments
                              └─ vault
                                   ├─ meeting_notes
                                   ├─ decisions
                                   ├─ retrospectives
                                   ├─ run_sheets ─── run_sheet_items

Key schema decisions

  • work_items.position — integer sort order within a stage column. Updated on every drag-end in a single transaction.
  • custom_stages.is_done — boolean flag marking the terminal "done" stage. Used to compute epic/sprint completion without hardcoding stage names.
  • wiki_pages.parent_id — self-referential FK enabling arbitrarily deep page trees. Null = root page.
  • app_meta table — key/value store for schema_version, active_user_id, last_drive_sync, drive_file_id.
  • Migrations run sequentially at startup, gated by app_meta.schema_version. Each migration is an idempotent SQL file using CREATE TABLE IF NOT EXISTS.

Google Drive Sync Strategy

Auth flow

OAuth2 PKCE — entirely in the browser, no backend. Scope: https://www.googleapis.com/auth/drive.appdata. Auth uses a popup (window.open) so the main window's COOP/COEP headers are never broken by a navigation. Token stored in localStorage (acceptable for a personal-device PWA with no shared sessions).

Backup file

A single file named planner-backup.sql in the Drive AppData folder. Contents: plain SQL INSERT statements generated by iterating all table rows. Uploaded via Drive REST API v3 multipart upload.

Sync operations

  • Export: Iterate all tables → build INSERT SQL → PUT to Drive. Timestamp saved to app_meta.last_drive_sync.
  • Import: Download SQL from Drive → DELETE all local rows (in FK-safe order) → db.exec(sql) → reload app state.
  • Auto-sync: setInterval every 15 minutes when navigator.onLine. Triggered in useGoogleDrive hook on mount.

Conflict model (v1)

Last-write-wins. The updatedTime of the Drive file is compared with app_meta.last_drive_sync. If Drive is newer, user is prompted to import. If local is newer, export overwrites Drive. Two concurrent active sessions will lose one device's changes — acceptable for a solo/small-team tool. v2 will track per-row vector clocks.

Roles & Permissions

There is no auth server. "Authentication" is identity — the current user is stored in app_meta.active_user_id. Role enforcement is application-layer only (appropriate for a local-data, personal-use tool).

PWA Strategy

Cross-Origin Isolation (COOP/COEP)

PGlite requires SharedArrayBuffer, which requires Cross-Origin Isolation. In dev, Vite serves COOP/COEP headers directly. In production (any static host), coi-serviceworker.js — loaded as the very first script in index.html — installs a service worker that patches these headers on every navigation response. This avoids needing host-level header configuration on Netlify, Vercel, or GitHub Pages.

Service worker

Generated by vite-plugin-pwa using Workbox. Strategy: registerType: 'autoUpdate' — the SW updates silently in the background and activates on next navigation. navigateFallback: '/index.html' for SPA routing. The /docs/ path is excluded from the navigation fallback so static HTML docs are served directly.

Manifest

public/manifest.webmanifest declares display: standalone, theme/background colors matching the dark UI, and 192px + 512px icons. The 512px icon has purpose: "any maskable" for Android adaptive icons.
